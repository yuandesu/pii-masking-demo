Redact PII from APM traces at the Datadog Agent layer — before data leaves your infrastructure. Zero app code changes.

🎯Purpose

This demo proves that the Datadog Agent can intercept and redact PII (Personally Identifiable Information) from APM traces using replace_tags — without touching a single line of application code.

The core problem: Error messages often include user data. When an error says "User john.doe@company.com was not found", that email gets recorded in the APM trace span's error.message tag. If the trace reaches Datadog's backend, the PII is now in a third-party system — a GDPR/CCPA compliance risk. The Agent's replace_tags config applies regex substitution to all traces before forwarding them. Your application code never needs to change.

Two identical Node.js apps, two Datadog Agents. Port 3004 routes through the masked agent (replace_tags active). Port 3005 routes through the raw agent (no masking). Trigger the same error on both ports and compare what reaches Datadog's backend.

💡Key Concepts

  • replace_tags: An Agent-side config block that rewrites span tag values using regex. Applied to all spans before they leave your infrastructure.
  • name: "*": The wildcard that matches all span tags — including error.message, http.url, user.email, and any custom tag.
  • The raw PII never reaches Datadog's servers when masking is enabled — the regex runs inside your own Agent, on your own infrastructure.
  • dd-trace (Node.js): Datadog's Node.js APM library. Auto-instruments HTTP, Express, databases, and more.
  • Zero code changes: Masking is 100% in Agent config. No application deployments required to enable or tune the regex.

🚀Quick Start

  1. Copy env file: 
    cp .env.example .env
  2. Set your API key in .env: 
    DD_API_KEY=your_key_here
  3. Start all services (unset local DD_API_KEY to avoid conflict): 
    env -u DD_API_KEY docker compose up -d --build
  4. Trigger a masked error: 
    curl -X POST http://localhost:3004/lookup   -d '{"email":"john.doe@company.com"}'
  5. Trigger a raw error: 
    curl -X POST http://localhost:3005/lookup   -d '{"email":"john.doe@company.com"}'

在 Datadog Agent 層從 APM 追蹤中刪除 PII，在資料離開您的基礎設施之前——零應用程式程式碼變更。

🎯目的

本示範證明 Datadog Agent 可以使用 replace_tags 攔截並刪除 APM 追蹤中的 PII（個人識別資訊），無需更改任何應用程式程式碼。

核心問題：錯誤訊息通常包含使用者資料。當錯誤顯示「User john.doe@company.com was not found」時，該電子郵件會記錄在 APM 追蹤 span 的 error.message 標籤中。如果追蹤到達 Datadog 後端，PII 就進入了第三方系統——這是 GDPR/CCPA 合規風險。Agent 的 replace_tags 設定在轉發追蹤之前在代理內部應用正規表達式替換，您的應用程式程式碼永遠不需要更改。

💡核心概念

  • replace_tags：使用正規表達式重寫 span 標籤值的 Agent 端設定塊，在所有 span 離開基礎設施之前應用。
  • name: "*"：匹配所有 span 標籤的萬用字元，包括 error.message、http.url、user.email 和任何自訂標籤。
  • 啟用遮罩時，原始 PII 永遠不會到達 Datadog 伺服器——正規表達式在您自己的 Agent 內部運行。
  • dd-trace (Node.js)：Datadog 的 Node.js APM 函式庫，自動插樁 HTTP、Express、資料庫等。
  • 零程式碼變更：遮罩 100% 在 Agent 設定中。啟用或調整正規表達式無需重新部署應用程式。
